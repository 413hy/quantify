# V4.7 策略与持仓接管复审（2026-07-15）

## 结论

V4.6 的信号生成、方向化预测限价和原生保护符合既有 Testnet 文档，但活动仓位被确认层排除，
不符合“以最新交易信号为主”的新增要求。V4.7 保留单币种唯一订单所有者，通过受确认门控的
控制邮箱实现同向加仓和反向换向；未确认信号不能覆盖仓位。

V4.7 仍是 `UNVALIDATED_TESTNET_EXPERIMENT`。该修改修复的是信号路由和仓位生命周期，不证明
策略盈利，也不把 Testnet 结果外推到生产。

## 13:30（Asia/Shanghai）证据复核

归档 `20260715-v45-final` 的 UTC 05:27–05:34:59（北京时间 13:27–13:34）显示：

- BTC 区间变化约 +0.015 bps、BNB 约 -5.09 bps，没有形成有效做多计划；
- ETH、SOL、XRP 分别上涨约 +11.65、+20.61、+18.19 bps，但五币宽度和本币确认没有在
  13:30 当时同时完成；
- XRP、SOL 的持续宽度做多计划分别在 13:34:15 和 13:34:26 才成立；系统于 13:34:26
  提交两张预测限价单，两张都在 13:35:18 因未回撤到限价而撤销。

所以这段不是服务停止，而是“确认发生在行情后段，加上只等回撤的被动限价未成交”。V4.7
不删除质量门控，也不把所有上涨都倒推成当时可交易信号；它解决的是已有仓位仍能接收后来确认的
同向/反向信号。入场延迟和预测限价成交率仍需独立样本优化。

## 需求对照

| 需求 | V4.7 实现 |
| --- | --- |
| 最新信号优先 | 仅最新且完整通过确认门控的计划可投递；邮箱消费时丢弃更旧计划 |
| 同向可加仓 | 预测 GTX 加仓；同一连续信号事件仅一次；未成交不追单 |
| 加仓后重设止盈止损 | 读取整仓数量与加权入场，撤旧保护并重挂整仓原生保护 |
| 反向平旧开新 | 先撤旧保护和平旧仓，记录 `SIGNAL_REVERSAL`，随后直接提交已确认新方向 |
| 防止无限叠仓 | 整仓 1 USDT 净损失预算、交易所最小数量、每日次数/亏损门槛和信号事件去重 |
| Codex 不在线仍交易 | systemd 的确定性规则服务独立运行，`codex_dependency=false` |
| 无持仓时间到期退出 | 保持禁用；仅原生止盈、止损、有效反向信号或操作员停止退出 |

## 剩余风险

- 加仓成交与替换保护之间存在很短的 REST 往返窗口；执行器在替换失败时 fail-closed 平仓，并且
  挂单期间保留旧保护，但 Binance 接口不提供把加仓和两张新保护组成单个原子事务的能力；
- 反向旧仓使用市价退出以确保方向及时终止，新方向仍用预测限价，可能出现旧仓已平而新仓未成交；
- 13:30 案例说明当前宽度确认可能晚于单币快速行情，2–8 bps 被动回撤也会降低成交率；应按
  `TESTNET_PREDICTIVE_LIMIT_SUBMITTED` 的等待时长、未成交率、成交后净结果分别复核，不能只按
  K 线事后上涨判断漏单；
- 至少积累 30 个 V4.7 已完成仓位生命周期，并单独统计加仓、反向、普通入场三类结果后，才能
  对门控、预测距离或止盈止损参数提出数据驱动调整。
